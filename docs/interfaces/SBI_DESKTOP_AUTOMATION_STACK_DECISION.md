# SBI 桌面自动化技术选型表（阶段决策版）

本文用于固化 SBI 桌面交易自动化的当前技术选型判断，作为后续实施、复盘与重构时的统一参考。

结论先行：

- Phase 1 不做正式 App，先做可审计、可回放、可人工确认的执行核心。
- Phase 1 核心执行层使用 Python。
- 视觉层不直接照搬 MaaEnd；我们采用 `UI Automation / Win32` 主路径，`OpenCV + OCR` 兜底。
- 控制层采用控制器抽象，但主执行手段应是 `UIA + Win32`，不是纯图像驱动点击。
- Phase 2 再补正式 App，优先方案为 `Tauri App + Python Core`。

---

## 1. 场景定义

目标系统不是游戏助手，而是 Windows 上的券商桌面端执行器。核心问题不是极限性能，而是：

- 状态识别是否可靠
- 执行流程是否可控
- 异常恢复是否清晰
- 审计与回放是否完整
- 是否能先从“半自动 + 人工确认”安全上线

因此，本项目的技术选型优先级为：

1. 可靠性
2. 可审计性
3. 与现有 J-stock Python 主链路的集成成本
4. 演进到正式 App 的可行性
5. 开发速度
6. 单次动作性能

## 3. 核心执行层选型

### 3.1 候选对比

| 方案 | 适配度 | 优点 | 问题 | 结论 |
| --- | --- | --- | --- | --- |
| Python | 高 | 与 J-stock 现有链路一致；Windows 自动化与视觉生态成熟；适合快速做审计、回放、风控与半自动流程 | 原生桌面产品感弱于 C#；底层消息控制不如 C#/C++ 直接 | 最优方案 |
| C# | 中高 | Windows 原生能力最强；长期产品化优秀；桌面 UI 与打包体验好 | 与现有 Python 策略链路割裂；第一阶段原型速度慢 | Phase 2 壳层候选，不做核心主线 |
| Go | 中低 | 单文件部署、并发体验好 | Windows 自动化生态不占优；UIA/CV/OCR/业务编排整体不如 Python/C# | 不采用 |
| C++ | 低 | 局部算法性能好 | 不适合做整套业务编排；开发成本高 | 不采用 |

### 3.2 最终判断

核心执行层选择 Python，不是因为个人偏好，而是因为它在本项目里提供了最佳工程平衡：

- 能直接接入现有 Python 信号与执行建议链路
- 能最快实现 `dry-run / replay / screenshots / structured logs / audit trail`
- 能同时整合 `UIA / Win32 / OCR / OpenCV / SQLite`
- 适合先落地“人工确认后提交”的安全模式

### 3.3 不把 C# 作为第一阶段核心的原因

C# 只在一个前提下可能优于 Python：

- 我们已经确认产品最终形态会长期重度依赖 Windows 原生桌面能力

但在当前阶段，真正的难点不是 UI，而是执行可靠性。若过早把核心迁到 C#，会明显增加原型成本，却不直接提高第一阶段最关键的成功概率。

因此：

- 当前最佳方案：`Python Core`
- 未来若 App 复杂度显著上升：保留 `C# Frontend + Python Core` 作为二线方案

## 4. 任务编排层选型

### 4.1 决策

采用“配置化状态机 + Python 执行器”：

- 执行任务定义放在 JSON 或 YAML
- Python 负责调度、状态转移、校验、异常处理
- 难点识别与特殊规则再下沉到专用模块

### 4.2 原因

SBI 执行流程天然就是状态机：

- 启动客户端
- 登录或会话恢复
- 切换账户或目标页面
- 输入证券代码
- 输入数量/价格/订单方式
- 校验确认页
- 最终提交或等待人工确认

这类流程如果完全硬编码在业务函数里，后续维护成本会迅速恶化。配置化编排更适合长期演进。

### 4.3 与 MaaEnd 的关系

这一层最值得借鉴 MaaEnd。

MaaEnd 主体采用：

- `Pipeline JSON` 定义任务流
- Go service 处理复杂逻辑
- 仅在少数识别难点使用 `cpp-algo`

本项目借鉴的方法是：

- 学它的“流程配置化”
- 不直接照搬其 `Pipeline / PI 协议 / Go service` 技术栈

## 5. 视觉识别层选型

### 5.1 MaaEnd 的视觉层是什么

MaaEnd 的视觉层核心来自 MaaFramework，其公开信息表明它主要依赖：

- 截图驱动的状态识别
- 模板/颜色/场景识别
- `OpenCV`
- `FastDeploy`
- `ONNX Runtime`
- `DirectML`

也就是说，MaaEnd 的主路径是“图像识别驱动自动化”。这非常适合游戏、模拟器、Unity 窗口等高图像依赖场景。

### 5.2 我们的视觉层决策

本项目不应与 MaaEnd 完全一样。

本项目视觉层采用：

- 主路径：`UI Automation / Win32 控件识别`
- 兜底：`OpenCV + OCR`
- 可选增强：`ONNX Runtime` 仅用于少量难识别页面

### 5.3 选型判断表

| 方案 | 角色 | 结论 | 原因 |
| --- | --- | --- | --- |
| UI Automation | 主路径 | 采用 | SBI 是桌面业务软件，不是纯图像界面；若能读控件，就不应退化为看图点按钮 |
| Win32 窗口信息 | 主路径补充 | 采用 | 用于窗口定位、焦点、状态辅助与坐标映射 |
| OpenCV | 兜底 | 采用 | 适合做图标、区域、弹窗与非标准控件识别 |
| OCR | 兜底 | 采用 | 适合做确认页文本校验、错误提示识别、成交信息提取 |
| ONNX 视觉模型 | 少量增强 | 保留 | 只在 OCR/模板法难以稳定覆盖时考虑 |
| 纯图像识别主驱动 | 主路径 | 不采用 | 对桌面业务软件不够稳，也不利于审计与定位 |

### 5.4 最终判断

MaaEnd 与我们的共同点是：

- 都必须在每一步动作前重新识别状态
- 都不能依赖盲点和固定 sleep

但差异也很明确：

- MaaEnd：视觉识别是主干
- 我们：控件识别是主干，视觉识别是兜底和校验

这不是能力倒退，而是针对不同目标软件形态做的正确选型。

## 6. 控制层选型

### 6.1 MaaEnd 的控制层是什么

MaaEnd 通过 MaaFramework 的控制器抽象提供不同控制方式，公开配置中可见：

- `Win32`
- `ADB`
- `PlayCover`
- Win32 前台/后台等不同模式

它的控制层本质是“控制器抽象 + 目标环境适配”。这一点值得直接借鉴。

### 6.2 我们的控制层决策

本项目控制层采用分层控制：

1. `UI Automation` 动作优先
2. `Win32` 负责窗口管理、定位与辅助控制
3. 键盘/鼠标注入作为兜底
4. 纯坐标点击只作为最后 fallback

### 6.3 选型判断表

| 控制方式 | 角色 | 结论 | 原因 |
| --- | --- | --- | --- |
| UIA 控件操作 | 主路径 | 采用 | 可读取控件类型、名称、焦点、可用状态，最适合业务软件 |
| Win32 窗口控制 | 主路径补充 | 采用 | 适合窗口枚举、激活、区域获取、消息辅助 |
| 键盘/鼠标注入 | 兜底 | 采用 | 用于少量无法直接通过 UIA 完成的动作 |
| 纯坐标点击 | 最后 fallback | 保留 | 可用，但不应成为主路径 |
| ADB / 模拟器控制 | 非目标路径 | 不采用 | 当前目标是 Windows 桌面券商端，不是移动端主战场 |

### 6.4 最终判断

我们与 MaaEnd 在控制层的共同点是：

- 都应该有控制器抽象
- 都应该允许不同执行后端存在

但主执行手段不同：

- MaaEnd 更偏“图像识别后执行控制器动作”
- 我们更偏“控件驱动执行，图像识别辅助确认”

## 7. 数据、审计与安全层选型

### 7.1 决策

Phase 1 采用：

- `SQLite` 保存任务状态、执行步骤、异常、人工确认结果
- 结构化日志保存每一步输入、识别结果、决策理由
- 截图或局部裁剪图保存关键节点证据

### 7.2 原因

券商自动化不是单纯“能点成功”就够了，必须能回答：

- 当时看到了什么
- 为什么做这个动作
- 提交前校验过什么
- 为什么没有继续执行

因此，审计层是核心层的一部分，不是附属功能。

## 8. App 层选型

### 8.1 当前决策

Phase 1 不做正式 App。

理由：

- 当前最关键的是执行可靠性，不是产品包装
- 先做 CLI + 调试面板，能更快验证识别、控制、审计链路
- 若先做 App，容易把实现重点从“能否安全执行”错误转移到“界面是否完整”

### 8.2 Phase 2 目标

正式 App 优先方案：`Tauri App + Python Core`

原因：

- Tauri 适合作为轻量桌面壳
- 前端可用 React/TypeScript 做配置、审计查看、执行面板
- Python Core 继续保留，不破坏核心执行链路

### 8.3 备选方案

备选：`C# App + Python Core`

适用条件：

- 确认需要更重的 Windows 原生交互
- 长期桌面产品化要求高于跨平台与前端开发效率

### 8.4 不推荐方案

不推荐在当前阶段直接做：

- `C# 全栈重写`
- `Go 全栈重写`
- `直接套 MaaFramework / MaaEnd 作为主框架`

原因是这些路线都会显著提高第一阶段复杂度，但不直接提升第一阶段成功率。

## 10. 明确不采用的路线

以下路线当前不作为主线：

1. 纯图像识别主驱动
2. 纯坐标点击主驱动
3. MaaFramework / MaaEnd 全量照搬
4. Go 作为主执行层
5. C++ 作为主执行层
6. 第一阶段先做正式 App

---

## 11. 后续复用原则

后续所有实现讨论默认以本文为基线：

- 若只讨论识别与控制细节，不重新争论主语言
- 若新增视觉模块，默认它是 UIA/Win32 的补充，不反客为主
- 若讨论 App，默认不影响 Python Core
- 若要偏离本文结论，必须说明新的约束变化，例如：
  - SBI 客户端版本变化导致 UIA 完全不可用
  - 最终产品目标变成长期 Windows 原生重产品
  - 某类确认页必须依赖模型识别且 OCR 无法稳定覆盖

---

## 12. 一句话结论

这套系统的最佳方案不是“像 MaaEnd 那样做一个图像驱动助手”，而是“借鉴 MaaEnd 的分层与方法论，用 Python 做核心执行器，以 UIA + Win32 为主、OpenCV/OCR 为辅，并把正式 App 放到第二阶段”。
